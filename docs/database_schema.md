# Database Schema Documentation

## Overview

This document describes the PostgreSQL database schema for the Kayak project. The schema consists of two main tables: `destinations` and `hotels`, with a one-to-many relationship.

## Tables

### 1. destinations

Stores city information with aggregated weather data.

| Column                | Type          | Constraints         | Description                                |
|-----------------------|---------------|---------------------|--------------------------------------------|
| city_id               | INTEGER       | PK, AUTO_INCREMENT  | Auto-generated unique identifier           |
| city_name             | VARCHAR(100)  | UNIQUE, NOT NULL, INDEX | City name (natural key from CSV)     |
| latitude              | FLOAT         | NOT NULL            | City latitude coordinate                   |
| longitude             | FLOAT         | NOT NULL            | City longitude coordinate                  |
| weather_score         | FLOAT         | NOT NULL, INDEX     | Computed weather score for ranking         |
| avg_temperature_7d    | FLOAT         | NULL                | Average temperature over 7-day forecast    |
| avg_rain_probability  | FLOAT         | NULL                | Average rain probability (0-100)           |
| avg_cloud_coverage    | FLOAT         | NULL                | Average cloud coverage (0-100)             |
| forecast_count        | INTEGER       | NULL                | Number of forecast data points             |

**Indexes:**
- `city_name`: For fast city lookups (unique constraint creates implicit index)
- `weather_score`: For sorting destinations by weather quality

**Business Rules:**
- `city_name` is the natural key used in CSV files
- `city_id` is generated by PostgreSQL after insertion
- Weather aggregates can be NULL if computation fails
- Coordinates are required for all destinations

### 2. hotels

Stores hotel information linked to destinations.

| Column      | Type          | Constraints         | Description                              |
|-------------|---------------|---------------------|------------------------------------------|
| hotel_id    | INTEGER       | PK, AUTO_INCREMENT  | Auto-generated unique identifier         |
| city_id     | INTEGER       | FK, NOT NULL, INDEX | Reference to destinations.city_id        |
| hotel_name  | VARCHAR(255)  | NOT NULL            | Hotel name                               |
| url         | TEXT          | NOT NULL            | Booking URL (can be long)                |
| score       | FLOAT         | INDEX               | Hotel rating/score                       |
| address     | TEXT          | NULL                | Hotel address (free text)                |
| description | TEXT          | NULL                | Hotel description text                   |
| latitude    | FLOAT         | NULL                | Hotel-specific latitude (geocoded)       |
| longitude   | FLOAT         | NULL                | Hotel-specific longitude (geocoded)      |

**Indexes:**
- `city_id`: For fast joins with destinations table
- `score`: For sorting hotels by rating

**Business Rules:**
- `city_id` foreign key enforces referential integrity
- Hotels have individual GPS coordinates (geocoded from address)
- `url` is TEXT to accommodate long booking URLs
- Multiple hotels can have the same name in different cities

## Relationships

### destinations → hotels (1:N)

- **Type:** One-to-Many
- **Foreign Key:** `hotels.city_id` → `destinations.city_id`
- **Cascade:** Not specified (default behavior - prevent deletion of destination with hotels)
- **SQLAlchemy:** Bidirectional relationship via `back_populates`

**Usage:**
```python
# Access hotels from destination
destination = session.query(Destination).filter_by(city_name="Paris").first()
hotels = destination.hotels  # List of Hotel objects

# Access destination from hotel
hotel = session.query(Hotel).first()
city = hotel.destination  # Destination object
```

## Data Flow

### ETL Process (CSV → PostgreSQL)

1. **Destinations:**
   - Load `top_destinations.csv`
   - Insert cities with `city_name` (natural key)
   - PostgreSQL generates `city_id` automatically
   - Retrieve generated `city_id` for each city

2. **Hotels:**
   - Load `top_hotels.csv` (contains `city_name`, not `city_id`)
   - Map `city_name` → `city_id` using destinations table
   - Insert hotels with `city_id` foreign key

### Key Constraint

**CSV files do NOT contain city_id.** The ETL process must:
1. Insert destinations first
2. Query back the generated `city_id` for each city
3. Use these `city_id` values when inserting hotels

## Index Strategy

Indexes are placed on columns used for:

1. **Lookups:** `city_name` (unique searches)
2. **Sorting:** `weather_score`, `score` (ORDER BY queries)
3. **Joins:** `city_id` in hotels (foreign key)

**Performance Considerations:**
- Indexes speed up SELECT but slow down INSERT
- For initial bulk load, consider disabling indexes during ETL
- Small dataset (<1000 cities, <20000 hotels) makes indexes less critical

## Data Types Rationale

- **FLOAT vs NUMERIC:** Using FLOAT for simplicity and speed (KISS principle)
  - Sufficient precision for GPS coordinates (±6 decimals)
  - Adequate for scores and weather metrics
  - Faster than NUMERIC for our use case

- **VARCHAR vs TEXT:**
  - Fixed length for predictable fields (city_name, hotel_name)
  - TEXT for variable length (url, address)

- **INTEGER:** Standard for IDs and counts

## SQL DDL (Generated by SQLAlchemy)

```sql
CREATE TABLE destinations (
    city_id SERIAL PRIMARY KEY,
    city_name VARCHAR(100) UNIQUE NOT NULL,
    latitude FLOAT NOT NULL,
    longitude FLOAT NOT NULL,
    weather_score FLOAT NOT NULL,
    avg_temperature_7d FLOAT,
    avg_rain_probability FLOAT,
    avg_cloud_coverage FLOAT,
    forecast_count INTEGER
);

CREATE INDEX ix_destinations_city_name ON destinations(city_name);
CREATE INDEX ix_destinations_weather_score ON destinations(weather_score);

CREATE TABLE hotels (
    hotel_id SERIAL PRIMARY KEY,
    city_id INTEGER NOT NULL REFERENCES destinations(city_id),
    hotel_name VARCHAR(255) NOT NULL,
    url TEXT NOT NULL,
    score FLOAT,
    address TEXT,
    description TEXT,
    latitude FLOAT,
    longitude FLOAT
);

CREATE INDEX ix_hotels_city_id ON hotels(city_id);
CREATE INDEX ix_hotels_score ON hotels(score);
```
